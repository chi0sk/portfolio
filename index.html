02 / Work

Modules

001

NetReplication

A server-to-client state replication system built around named data containers called replicas. The server owns the data, clients subscribe by token and receive batched updates on a configurable tick rate. Handles delta compression, sequence gap detection, xxHash64 integrity verification, and automatic client-side rollback on corruption. If a client detects a gap or hash mismatch it requests a full state refresh without you having to do anything. Includes a simulation mode for testing with artificial latency, packet loss, and corruption.

-- server sets a value, clients receive it automatically replica:Set({ "coins" }, 500) -- client reads it print(replica:Get({ "coins" })) -- 500
Roblox Module 2025
002

ChronicleStore

A DataStore wrapper built around lease-based session ownership. Before loading data it acquires a distributed lease stored alongside the record. If another server holds an active lease it backs off. If the lease is expired it steals it. Saves use UpdateAsync with generation and ownership checks so stale writes are rejected at the datastore level, not just caught after the fact. Includes FNV-1a integrity hashing, exponential backoff, a circuit breaker, write coalescing, automatic lease renewal, and a simulation mode that works in Studio without API access enabled.

-- load a player's data, change it, save it local profile = Store:StartSessionAsync("player_123") profile.Data.coins += 100 profile:Save()
Roblox Module 2025
003

TaskScheduler

A distributed task queue with priority scheduling, dependency graphs, exactly-once execution, and cross-server work stealing. Tasks are submitted with a name, priority, payload, retry policy, and optional dependencies. Workers register handlers by task name and pull from a priority queue. If a server is underloaded it can request tasks from overloaded servers via MessagingService. Exactly-once mode uses a DataStore distributed lock to prevent two servers running the same task concurrently. Tasks that exhaust their retry budget land in a dead letter queue with full metadata.

-- queue a job, a worker picks it up and runs it scheduler:SubmitTask({ Name = "SendReward", Payload = { userId = 123 } }) -- somewhere else, a worker handles it Handlers.SendReward = function(task) giveCoins(task.Payload.userId) return true end
Roblox Module 2025
004

RateLimiter

Four rate limiting algorithms in one module - sliding window, token bucket, fixed window, and leaky bucket. Each has a different enforcement shape and suits different use cases. The token bucket uses a g-counter CRDT so state can be safely merged across servers without coordination rounds. Distributed sync goes over MessagingService with adaptive batching and rate cap awareness. Includes per-user global ceilings, violation tracking with time-based decay, escalating penalty windows, and LRU compaction when bucket count exceeds a configured threshold.

-- allow 5 purchases per minute per player, block if exceeded local result = limiter:Check(player.UserId, "purchase") if not result.Allowed then return "slow down" end
Roblox Module 2025
005

StateMachine

A hierarchical finite state machine with nested states, wildcard event routing, timed transitions, guard functions, and per-transition action lists. Events bubble from leaf to root so parent states can catch things their children do not handle. Sibling state names resolve relative to the current parent, so you can write "dead" instead of "combat.dead" inside a combat state. Snapshot and restore let you serialise the full machine into ChronicleStore and reload it later. Clean backbone for NPC behaviour, combat systems, UI flows, or anything else with distinct modes.

-- send events to move between states machine:Send("AGGRO") print(machine:GetState()) -- "combat.alive" machine:Send("KILLED") print(machine:GetState()) -- "combat.dead", auto-resets after 5s
Roblox Module 2025
006

CircuitBreaker

Wraps any fallible operation and stops calling it when it is consistently failing. Three states - closed for normal operation, open for fast-failing without calling the wrapped function, and half-open for probing whether the dependency has recovered. Opens on a consecutive failure threshold or on a sliding window failure rate. In half-open mode a configurable number of probe calls are let through. If enough succeed it closes, if any fail it reopens immediately. Includes per-call timeouts, fallback callbacks, EventBus integration, per-key config overrides, and stale circuit eviction.

-- if the DataStore keeps failing, stop hitting it local result = cb:Execute("leaderboard", function() return DataStore:GetAsync("top10") end) if result.Ok then renderBoard(result.Value) end
Roblox Module 2025
007

EventBus

A publish-subscribe event system backed by a trie for efficient routing with wildcard support. Patterns support exact matches, single-segment wildcards (*), and multi-segment wildcards (**). Subscribers are ordered by priority within each trie node. Middleware runs before dispatch and can transform or block events. Namespaces let you scope event names to a prefix without a separate bus instance. Distributed mode batches events and syncs them across servers via MessagingService with rate cap awareness and exponential backoff. Unhandled events are retained in a dead letter queue.

-- listen for an event bus:On("player.died", function(payload) print(payload.name, "died") end) -- fire it from anywhere bus:Emit("player.died", { name = "sam" })
